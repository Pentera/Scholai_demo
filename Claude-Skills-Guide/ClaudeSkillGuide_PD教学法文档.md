#### OBJ-01：理解技能核心概念与原则

> **目标内容**：学生将能够解释 Skill 的基本构成，并归纳其核心设计原则。

**教学行为**：
作者首先以定义开篇，将 Skill 拆解为明确的文件结构（YAML + Markdown + scripts），接着提出“渐进式披露”等核心原则。为了让原本抽象的技术架构易于理解，作者针对 MCP 构建者专门引入了“厨房类比”，将工具能力与领域知识进行了生动的分离映射，帮助学习者建立直观认知。

**回忆线索**：
- **[类比锚点]** 将 MCP 比作“提供工具和原料的专业厨房”，将 Skill 比作“菜谱”，因为它们共享“基础设施+使用指南”的特征。
- **[判定口诀]** 看到“三个层级”，立刻想到“渐进式披露（System Prompt -> Body -> Linked files）”。
- **[易错警示]** 警告“直接将所有指令塞给 Claude”，正确做法是“通过渐进式披露按需加载”。

---

#### EX-01：厨房类比模型

> **题目内容**：MCP provides the professional kitchen: access to tools, ingredients, and equipment. Skills provide the recipes: step-by-step instructions on how to create something valuable.

**讲解过程**：
作者在讲解 MCP 与 Skill 的协同关系时直接抛出该类比。通过对比“没有技能时用户的迷茫”和“有技能时自动激活工作流”，强调了提供“菜谱（知识层）”的重要性。

**强调点**：
- "MCP provides the professional kitchen... Skills provide the recipes"

**与其他示例的关联**：
无

**回忆线索**：
- **[类比锚点]** 将 MCP 比作“厨房”，因为它们共享“提供原材料和工具”的特征。
- **[类比锚点]** 将 Skill 比作“菜谱”，因为它们共享“提供分步指令”的特征。

#### OBJ-02：规划与设计技能场景

> **目标内容**：学生将能够概述具体的用例触发条件与步骤，并区分该用例属于哪种常见类别。

**教学行为**：
作者要求在写代码前先明确 2-3 个具体用例。为了具象化，给出了一个优秀的 Sprint Planning 用例定义，并总结了官方观察到的三大典型分类（资产生成、工作流自动化、MCP增强）。随后，给出了衡量技能是否达标的定量与定性指标框架。

**回忆线索**：
- **[判定口诀]** 看到“写代码前”，立刻想到“先定义触发词、步骤和预期结果”。
- **[类比锚点]** 将“分类1：资产生成”比作“无外部工具的内置魔法”，因为它们共享“仅依赖 Claude 原生能力”的特征。
- **[易错警示]** 警告“没有明确衡量标准”，正确做法是“结合 API 失败率等定量指标和用户干预次数等定性指标”。

---

#### EX-02：项目 Sprint 规划用例

> **题目内容**：Use Case: Project Sprint Planning
Trigger: User says "help me plan this sprint" or "create sprint tasks" ...

**讲解过程**：
作为“Good use case definition”的代表，作者将其拆解为 Trigger、Steps 和 Result 三个模块，示范了什么是具体可执行的用例描述。

**强调点**：
- "Trigger: User says 'help me plan this sprint'"

**与其他示例的关联**：
是 EX-03 中“工作流自动化”类别的具体实例。

**回忆线索**：
- **[判定口诀]** 看到“用例定义”，立刻想到“必须包含 Trigger、Steps 和 Result”。

---

#### EX-03：三大技能类别实例

> **题目内容**：Category 1: frontend-design skill... Category 2: skill-creator skill... Category 3: sentry-code-review skill...

**讲解过程**：
直接列举 Anthropic 官方仓库中的真实案例作为三种类别的支撑，让学习者能对号入座自己要做的技能属于哪一类。

**强调点**：
- 强调这三种类别是 Anthropic 团队的“实战观察总结”。

**与其他示例的关联**：
无

**回忆线索**：
- **[判定口诀]** 看到“需要外部系统数据”，立刻想到“属于 Category 3: MCP Enhancement”。

---

#### EX-04：成功标准度量

> **题目内容**：Quantitative metrics: Skill triggers on 90% of relevant queries... Qualitative metrics: Users don't need to prompt...

**讲解过程**：
作者将成功标准划分为两类：能用数字追踪的（如 0 报错、X 个工具调用），以及看用户体验的（如无需纠正、结果一致），从而构建了一个完整的评估体系。

**强调点**：
- "0 failed API calls per workflow"

**与其他示例的关联**：
为后续 OBJ-04 的测试与迭代提供了靶子。

**回忆线索**：
- **[判定口诀]** 看到“评估技能好坏”，立刻想到“定量的 API 成功率与定性的用户免干预率”。

---

#### OBJ-03：编写标准化的技能文件

> **目标内容**：学生将能够执行规范的文件结构搭建，并使用正确的语法编写包含有效触发词的 YAML 描述和 Markdown 指令。

**教学行为**：
作者给出了极其严格的技术规范红线（如文件必须叫 SKILL.md，不可有空格）。在讲解最核心的 YAML Frontmatter 时，作者使用了“Good vs Bad”的正反对比法，直接指出模糊描述的危害。在 Markdown 主体部分，提供了一个可直接复制填空的模板结构。

**回忆线索**：
- **[判定口诀]** 看到“文件夹命名”，立刻想到“kebab-case，绝对不能有空格”。
- **[判定口诀]** 看到“写 Description”，立刻想到“公式：What it does + When to use it”。
- **[易错警示]** 警告“在 YAML 中使用 XML 标签 (< >)”，正确做法是“绝对禁止，因为这是系统提示词注入口，有安全限制”。

---

#### EX-05：YAML Frontmatter 编写示范

> **题目内容**：Good: Analyzes Figma design files... Bad: Helps with projects.

**讲解过程**：
并排展示优秀的描述和糟糕的描述。明确指出糟糕的描述错在“太宽泛、缺乏触发词”。通过正例展示必须包含用户可能会说的原话（如 "design specs"）。

**强调点**：
- "MUST include BOTH: What the skill does AND When to use it"

**与其他示例的关联**：
无

**回忆线索**：
- **[判定口诀]** 看到“Bad description”，立刻想到“Helps with projects 等废话”。
- **[易错警示]** 警告“描述里全是技术术语”，正确做法是“必须包含用户触发时会说的口语化短语”。

---

#### EX-06：Markdown 主体指令模板

> **题目内容**：# Your Skill Name \n ## Instructions \n ### Step 1... \n ## Examples...

**讲解过程**：
直接给出一套包含 Instructions, Examples, Troubleshooting 三大块的骨架代码，让学习者可以直接复制并替换内容。

**强调点**：
- 建议将详细参考文档移到 `references/` 下以保持主文件简洁。

**与其他示例的关联**：
无

**回忆线索**：
- **[判定口诀]** 看到“写 Markdown 主体”，立刻想到“三大块：步骤、案例、排错”。

#### OBJ-04：测试与迭代技能

> **目标内容**：学生将能够验证技能的触发准确率与功能完整性，并基于执行异常（过触发/欠触发）调整描述。

**教学行为**：
作者提出从单一任务切入测试的“Pro Tip”，随后拆解了测试的三大维度：触发测试（正负样本）、功能测试和性能对比。在遇到触发问题时，给出了非常具体的排错动作（欠触发加细节，过触发加负向描述）。

**回忆线索**：
- **[判定口诀]** 看到“欠触发（Undertriggering）”，立刻想到“补充更多具体的业务关键词”。
- **[易错警示]** 警告“一上来就做庞大的自动化测试集”，正确做法是“先在单个高难度任务上跑通并抽取成功经验”。

---

#### EX-07：技能触发测试用例

> **题目内容**：Should trigger: "Help me set up a new ProjectHub workspace"... Should NOT trigger: "What's the weather..."

**讲解过程**：
展示了如何构造正负样本进行边界测试，明确技能的触发边界。

**强调点**：
- 既要测试它该触发的时候触发，也要测试它不该触发的时候闭嘴。

**与其他示例的关联**：
无

**回忆线索**：
- **[类比锚点]** 将“负向触发测试”比作“安检的白名单测试”，因为它们共享“拒绝无关内容”的特征。

#### OBJ-05：应用高级技能设计模式

> **目标内容**：学生将能够根据复杂业务需求，运用合适的高级模式设计技能指令结构。

**教学行为**：
作者像提供“设计模式（Design Patterns）”字典一样，枚举了5种进阶的编排模式：串行流、跨系统传递、自我修复循环、决策树路由以及合规拦截前置。每一种模式都配有清晰的 Markdown 骨架示例。

**回忆线索**：
- **[判定口诀]** 看到“需要把上一步的结果传给下一步”，立刻想到“使用 Pattern 1 或 Pattern 2，明确定义参数传递逻辑”。
- **[判定口诀]** 看到“涉及高风险动作（如支付）”，立刻想到“使用 Pattern 5，在执行前插入领域知识拦截”。
- **[易错警示]** 警告“依赖自然语言让 LLM 自己判断分支”，正确做法是“提供明确的 If-Else 决策树（Pattern 4）”。

---

#### EX-08 至 EX-12（模式实例整合）

> **题目内容**：Pattern 1 (Sequential), Pattern 2 (Multi-MCP), Pattern 3 (Iterative), Pattern 4 (Context-aware), Pattern 5 (Domain-specific)

**讲解过程**：
通过直接展示伪代码/Markdown 结构提纲的方式，把抽象的架构理念具象化为可以照猫画虎的排版格式。

**强调点**：
- 强调这些是“从早期使用者中涌现出来的模式，而非死板的教条”。

**与其他示例的关联**：
这 5 个模式平行且互补，可以组合使用。

**回忆线索**：
- **[判定口诀]** 看到“初稿质量差”，立刻想到“引入 Pattern 3: Refinement Loop 自动化校验脚手架”。

## Part C：讲师教学常用语

### 标志性口头禅

由于这是一份技术文档，较少口头禅，但有明显的技术文档高频指示语：

| 用语 | 原话 | 典型使用情境 | AI 使用示例 |
|------|------|--------------|-------------|
| 强调重点 | "MUST include BOTH:" | 讲解 YAML 必须包含的要素时 | "在写描述时，记住文档里说的 MUST include BOTH：它做什么和何时使用。" |
| 纠正错误 | "Wrong -" / "Correct" | 在代码块或配置中展示防错示例时 | "这里的格式不对，像文档里标注的 Wrong 一样，你缺少了触发词。" |
| 总结收尾 | "Key techniques:" | 在每种模式或类别的最后进行要点提取时 | "让我们回顾一下这种模式的 Key techniques 都有哪些。" |

### 高频表达模式

| 场景 | 句式 | 使用频率 |
|------|------|----------|
| 对比区分 | "Good: [...] Bad: [...]" | 高 |
| 提示排障路径 | "Symptom: [...] Cause: [...] Solution: [...]" | 高 |
| 引出应用场景 | "Use when: [...]" | 高 |
